Q1 Use DDL Commands to Create and Modify Database Tables 16 Marks

Introduction

Data Definition Language (DDL) is a subset of SQL used to define, modify, and delete database structures such as tables, indexes, and schemas. DDL commands do not manipulate data directly — they operate on the structure (schema) of the database. The main DDL commands are CREATE, ALTER, DROP, TRUNCATE, and RENAME.

1. CREATE Command

The CREATE command is used to create a new table or database.

Syntax:

CREATE TABLE table_name (
    column1 datatype constraint,
    column2 datatype constraint,
    ...
);

Example – Create a Student Table:

CREATE TABLE Student (
    StudentID   INT PRIMARY KEY,
    Name        VARCHAR(50) NOT NULL,
    Age         INT,
    Department  VARCHAR(30),
    Email       VARCHAR(50) UNIQUE
);

Example – Create a Courses Table:

CREATE TABLE Courses (
    CourseID    INT PRIMARY KEY,
    CourseName  VARCHAR(100) NOT NULL,
    Credits     INT DEFAULT 3,
    StudentID   INT,
    FOREIGN KEY (StudentID) REFERENCES Student(StudentID)
);

2. ALTER Command

The ALTER command modifies an existing table structure — adding, modifying, or dropping columns.

-- a) Add a new column
ALTER TABLE Student ADD PhoneNumber VARCHAR(15);

-- b) Modify an existing column
ALTER TABLE Student MODIFY Age SMALLINT NOT NULL;

-- c) Drop a column
ALTER TABLE Student DROP COLUMN PhoneNumber;

-- d) Add a constraint
ALTER TABLE Student ADD CONSTRAINT chk_age CHECK (Age >= 17);

-- e) Rename a column (MySQL)
ALTER TABLE Student RENAME COLUMN Email TO EmailAddress;

3. DROP Command

Permanently removes a table and all its data from the database.

DROP TABLE Courses;
DROP DATABASE CollegeDB;
⚠️ DROP cannot be rolled back. All data is permanently deleted.

4. TRUNCATE Command

Removes all rows from a table but keeps the table structure intact.

TRUNCATE TABLE Student;
FeatureDROPTRUNCATE
Removes structureYesNo
Removes dataYesYes
Rollback possibleNoNo
FasterNoYes

5. RENAME Command

RENAME TABLE Student TO Students;
-- OR
ALTER TABLE Students RENAME TO StudentInfo;

Constraints Used in DDL

  • PRIMARY KEY – Uniquely identifies each row
  • FOREIGN KEY – Links two tables
  • NOT NULL – Field cannot be empty
  • UNIQUE – All values must be distinct
  • CHECK – Enforces a condition
  • DEFAULT – Assigns a default value
Conclusion: DDL commands form the backbone of database schema management. They allow DBAs and developers to precisely define the structure of the data to be stored, enforce integrity constraints, and evolve the schema over time without losing data unnecessarily.
Q2 Differentiate Entity Set and Attribute with Suitable Examples 16 Marks

Introduction

In the Entity-Relationship (ER) model, the two fundamental building blocks are Entities and Attributes. Understanding the difference is crucial for designing a good database schema.

Entity and Entity Set

An Entity is a real-world object or concept that has an independent existence and can be distinctly identified. An Entity Set is a collection of similar entities that share the same properties.

  • A Book in a library
  • A Bank Account in a bank

Q3 Demonstrate Database Operations – Creating a Table and Retrieving Records Using SQL 16 Marks

Introduction

SQL (Structured Query Language) is the standard language for interacting with relational databases. Database operations include creating tables (DDL), inserting data (DML), and retrieving records (DQL).

Step 1: Create the Database and Table

CREATE DATABASE CollegeDB;
USE CollegeDB;

CREATE TABLE Student (
    StudentID   INT PRIMARY KEY AUTO_INCREMENT,
    Name        VARCHAR(50) NOT NULL,
    Age         INT,
    Department  VARCHAR(30),
    CGPA        DECIMAL(3,2),
    City        VARCHAR(30)
);

Step 2: Insert Records (DML – INSERT)

INSERT INTO Student (Name, Age, Department, CGPA, City)
VALUES
('Alice',  20, 'CSE',  9.1, 'Chennai'),
('Bob',    21, 'ECE',  8.3, 'Mumbai'),
('Carol',  19, 'MECH', 7.5, 'Delhi'),
('David',  22, 'CSE',  8.8, 'Bangalore'),
('Eva',    20, 'IT',   9.4, 'Chennai'),
('Grace',  23, 'CSE',  7.2, 'Mumbai');

Step 3: Retrieve Records (DQL – SELECT)

-- a) All records
SELECT * FROM Student;

-- b) Specific columns
SELECT Name, Department, CGPA FROM Student;

-- c) With WHERE condition
SELECT * FROM Student WHERE Department = 'CSE';

-- d) Multiple conditions
SELECT * FROM Student WHERE Department = 'CSE' AND CGPA > 8.5;

-- e) ORDER BY
SELECT * FROM Student ORDER BY CGPA DESC;

-- f) LIMIT
SELECT * FROM Student ORDER BY CGPA DESC LIMIT 3;

-- g) Aggregate functions
SELECT Department, AVG(CGPA) AS Avg_CGPA FROM Student GROUP BY Department;
SELECT COUNT(*) AS Total_Students FROM Student;
SELECT MAX(CGPA) AS Highest, MIN(CGPA) AS Lowest FROM Student;

-- h) HAVING clause
SELECT Department, AVG(CGPA) AS Avg_CGPA
FROM Student
GROUP BY Department
HAVING AVG(CGPA) > 8.0;

Step 4: Update and Delete

UPDATE Student SET CGPA = 9.5 WHERE StudentID = 3;
DELETE FROM Student WHERE CGPA < 7.0;

Step 5: JOIN Query (Two Tables)

SELECT S.Name, S.Department, C.CourseName
FROM Student S
INNER JOIN Courses C ON S.StudentID = C.StudentID;

Conclusion: SQL provides a rich set of commands to create, populate, and query databases efficiently. Using SELECT with WHERE, ORDER BY, GROUP BY, HAVING, and JOINs enables precise data retrieval for real-world applications.
Q4 Apply DDL, DML, and DCL Commands in a Banking Application 16 Marks

Introduction

  • DDL (Data Definition Language) – Defines database structure (CREATE, ALTER, DROP)
  • DML (Data Manipulation Language) – Manipulates data (INSERT, UPDATE, DELETE, SELECT)
  • DCL (Data Control Language) – Controls access (GRANT, REVOKE)

DDL Commands – Creating Banking Tables

CREATE DATABASE BankDB;
USE BankDB;

CREATE TABLE Customer (
    CustomerID   INT PRIMARY KEY AUTO_INCREMENT,
    Name         VARCHAR(50) NOT NULL,
    Address      VARCHAR(100),
    Phone        VARCHAR(15) UNIQUE,
    Email        VARCHAR(50)
);

CREATE TABLE Account (
    AccountNo    INT PRIMARY KEY,
    CustomerID   INT,
    AccountType  VARCHAR(20) CHECK (AccountType IN ('Savings','Current')),
    Balance      DECIMAL(12,2) DEFAULT 0.00,
    OpenDate     DATE,
    FOREIGN KEY (CustomerID) REFERENCES Customer(CustomerID)
);

CREATE TABLE Transactions (
    TransID     INT PRIMARY KEY AUTO_INCREMENT,
    AccountNo   INT,
    TxnType     VARCHAR(10) CHECK (TxnType IN ('Credit','Debit')),
    Amount      DECIMAL(12,2),
    TxnDate     DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (AccountNo) REFERENCES Account(AccountNo)
);

DML Commands – Data Operations

-- INSERT
INSERT INTO Customer (Name, Address, Phone, Email)
VALUES ('Ravi Kumar', 'Chennai', '9876543210', 'ravi@email.com');

INSERT INTO Account (AccountNo, CustomerID, AccountType, Balance, OpenDate)
VALUES (1001, 1, 'Savings', 50000.00, '2023-01-15');

-- UPDATE: Deposit Rs. 10,000
UPDATE Account SET Balance = Balance + 10000 WHERE AccountNo = 1001;

-- UPDATE: Withdraw Rs. 5,000
UPDATE Account SET Balance = Balance - 5000
WHERE AccountNo = 1001 AND Balance >= 5000;

-- DELETE
DELETE FROM Customer WHERE CustomerID = 5;

-- SELECT with JOIN
SELECT C.Name, A.AccountNo, A.AccountType, A.Balance
FROM Customer C
JOIN Account A ON C.CustomerID = A.CustomerID
WHERE A.AccountNo = 1001;

-- Create users
CREATE USER 'bank_teller'@'localhost' IDENTIFIED BY 'Teller@123';

GRANT SELECT, INSERT ON BankDB.Transactions TO 'bank_teller'@'localhost';
GRANT ALL PRIVILEGES ON BankDB.* TO 'bank_manager'@'localhost';
GRANT SELECT ON BankDB.* TO 'auditor'@'localhost';

-- REVOKE permission
REVOKE DELETE ON BankDB.Transactions FROM 'bank_teller'@'localhost';

FLUSH PRIVILEGES;

Q5 What is Normalization? Explain 3NF, BCNF, and 4NF with Examples 16 Marks

What is Normalization?

Normalization is the process of organizing a relational database to reduce data redundancy and improve data integrity. It decomposes tables into smaller, well-structured tables following a set of rules called Normal Forms (NF).

Benefits of Normalization

  1. Eliminates redundancy – Avoids storing the same data multiple times
  2. Prevents update anomalies – One update doesn't require changing many rows
  3. Prevents insertion anomalies – Data can be inserted without unrelated data
  4. Prevents deletion anomalies – Deleting one fact doesn't lose another
  5. Improves performance – Smaller, focused tables are faster to query
  6. Ensures consistency – Single source of truth for each fact

Normal Forms Overview

Normal FormRequirement
1NFAtomic values, no repeating groups
2NF1NF + No partial dependencies
3NF2NF + No transitive dependencies
BCNF3NF + Every determinant is a candidate key
4NFBCNF + No multi-valued dependencies

Third Normal Form (3NF)

Definition: A relation is in 3NF if it is in 2NF and no non-prime attribute is transitively dependent on the primary key.

Transitive Dependency: A → B → C, where A is PK and C depends on A through B.

Example – Violation:

StudentIDNameZipCodeCity
101Alice600001Chennai
102Bob400001Mumbai

StudentID → ZipCode → City — City is transitively dependent. Violates 3NF.

After 3NF Decomposition:

Student(StudentID, Name, ZipCode)
ZipLocation(ZipCode, City)

Boyce-Codd Normal Form (BCNF)

Definition: For every functional dependency X → Y, X must be a superkey. BCNF is stronger than 3NF.

Example – Violation: CourseEnrollment(StudentID, Course, Instructor)

  • {StudentID, Course} → Instructor
  • Instructor → Course ← Instructor is NOT a superkey → violates BCNF

After BCNF Decomposition:

InstructorCourse(Instructor, Course)
StudentInstructor(StudentID, Instructor)

Fourth Normal Form (4NF)

Definition: A relation is in 4NF if it is in BCNF and has no multi-valued dependencies (MVD). If A →→ B, then for each value of A, there are multiple independent values of B.

Example – Violation: Employee(EmpID, Skill, Hobby)

EmpID →→ Skill and EmpID →→ Hobby independently → causes redundancy.

After 4NF Decomposition:

EmpSkill(EmpID, Skill)
EmpHobby(EmpID, Hobby)

Comparison: 3NF vs BCNF vs 4NF

Feature3NFBCNF4NF
Eliminates transitive dependencyYesYesYes
Every determinant is candidate keyNot requiredRequiredRequired
Handles multi-valued dependenciesNoNoYes
Lossless decompositionYesSometimesYes
Dependency preservationYesMay notYes
Conclusion: Normalization is essential for building a clean, efficient, and consistent database. 3NF removes transitive dependencies, BCNF strengthens it by requiring every determinant to be a candidate key, and 4NF eliminates multi-valued dependencies. Each level builds upon the previous to ensure higher data integrity.

Q7 ACID Properties – Examine and Implement in Transaction Processing 16 Marks

Introduction

ACID stands for Atomicity, Consistency, Isolation, and Durability. These four properties guarantee reliable transaction processing in a DBMS, especially under concurrent access and system failures.

a) Atomicity

Definition: A transaction is treated as a single unit — it either completely succeeds or completely fails. Any partial failure causes a full rollback.

START TRANSACTION;
UPDATE Account SET Balance = Balance - 5000 WHERE AccountNo = 1001;
UPDATE Account SET Balance = Balance + 5000 WHERE AccountNo = 1002;
COMMIT;   -- If both succeed
-- ROLLBACK;  If any step fails

b) Consistency

Definition: A transaction must bring the database from one valid state to another valid state, maintaining all defined constraints and integrity rules.

-- Balance cannot go below 0
ALTER TABLE Account ADD CONSTRAINT chk_balance CHECK (Balance >= 0);
-- Before: A=10000, B=5000, Total=15000
-- After transfer: A=7000, B=8000, Total=15000 ✓

c) Isolation

Definition: Concurrent transactions execute independently. The intermediate state of one transaction is not visible to others until committed.

SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
START TRANSACTION;
SELECT Balance FROM Account WHERE AccountNo = 1001;
COMMIT;
Isolation LevelDirty ReadNon-Repeatable ReadPhantom Read
READ UNCOMMITTEDPossiblePossiblePossible
READ COMMITTEDPreventedPossiblePossible
REPEATABLE READPreventedPreventedPossible
SERIALIZABLEPreventedPreventedPrevented

d) Durability

Definition: Once a transaction is committed, its changes are permanently saved — even if the system crashes immediately afterward.

-- Write-Ahead Logging (WAL)
-- Before change: write log entry to stable storage
-- On restart after crash: re-apply committed changes from log
START TRANSACTION;
UPDATE Account SET Balance = 7000 WHERE AccountNo = 1001;
COMMIT; -- Permanently persisted to disk

ACID Summary

PropertyGuaranteesMechanism
AtomicityAll-or-nothing executionROLLBACK / COMMIT
ConsistencyValid state before and afterConstraints, triggers
IsolationNo interference between transactionsLocks, MVCC
DurabilityCommitted data survives crashesWAL, checkpoints
Conclusion: ACID properties are the foundation of reliable database systems. Atomicity prevents partial updates, Consistency ensures rule compliance, Isolation handles concurrency, and Durability ensures data survival across failures.
Q8 Compare Conflict Serializability and View Serializability in DBMS 16 Marks

Introduction

In a multi-user DBMS, serializability ensures correctness of concurrent schedules. A schedule is serializable if its result equals some serial (sequential) execution. Two types exist: Conflict Serializability and View Serializability.

Conflict Serializability

Definition: A schedule is conflict serializable if it can be transformed into a serial schedule by swapping non-conflicting operations.

Conflicting operations belong to different transactions, access the same data, and at least one is a Write:

  • Read-Write (R-W) — conflict
  • Write-Read (W-R) — conflict
  • Write-Write (W-W) — conflict
  • Read-Read (R-R) — no conflict

Testing – Precedence Graph

Schedule: R1(A), W2(A), R1(B), W1(B), W2(B)

Conflicts:
  R1(A) before W2(A): T1 → T2  (R-W conflict)
  W1(B) before W2(B): T1 → T2  (W-W conflict)

Precedence graph: T1 → T2  (no cycle)
→ CONFLICT SERIALIZABLE
Equivalent serial schedule: T1, then T2

View Serializability

Definition: A schedule S is view serializable if it is view equivalent to some serial schedule S'.

Three conditions for view equivalence:

  1. Initial Read: If Ti reads the initial value of X in S, Ti reads initial value in S'
  2. Updated Read: If Ti reads value written by Tj in S, same in S'
  3. Final Write: If Ti performs the final write on X in S, Ti does so in S'
Schedule with blind write:
T1: W(A)
T2:      W(A)
T3:           R(A)

→ NOT conflict serializable (W-W then R depends on ordering)
→ IS view serializable (view equivalent to T1, T2, T3)

Comparison Table

FeatureConflict SerializabilityView Serializability
BasisConflict equivalenceView equivalence
Testing methodPrecedence graph (cycle detection)Match reads and final writes
ComplexityPolynomial O(n²)NP-complete
ScopeNarrowerBroader
Blind writesCannot handleCan handle
Practical useWidely used in real RDBMSTheoretically important
RelationshipIs always view serializableMay not be conflict serializable
Conclusion: Conflict serializability is the practical standard used in real systems — efficient to check via precedence graph. View serializability is broader but NP-complete to verify. Every conflict serializable schedule is view serializable, but not vice versa.
Q9 Examine Single-Level and Multi-Level Indexing for Large Databases 16 Marks

Introduction

Indexing is a data structure technique to quickly locate records without full table scans. It works like a book's index — jump directly to the relevant page rather than reading every page.

Single-Level Indexing

Definition: A single level of index entries pointing directly to data blocks on disk.

Types of Single-Level Indexes

1. Primary Index (on primary key — sparse):
   E001 → Block 1
   E004 → Block 2

2. Clustering Index (on non-key ordered field):
   CSE → Block 1
   ECE → Block 3

3. Secondary Index (on non-ordering field — dense):
   Alice → Block 3, Offset 2
   Bob   → Block 1, Offset 5

Limitations of Single-Level Indexing:

  • As database grows, the index file itself becomes very large
  • Searching a large index still requires many disk I/Os
  • Insertions/deletions require costly index reorganization

Multi-Level Indexing

Definition: Builds an index on top of the index, creating a hierarchy. Each higher level points to the level below.

Level 3 (Root Index)    ── 1 block
        ↓
Level 2 (Intermediate)  ── few blocks
        ↓
Level 1 (Base Index)    ── many blocks
        ↓
Data File               ── thousands of blocks

B+ Tree – Most Common Multi-Level Index

A self-balancing tree where internal nodes hold keys and pointers; leaf nodes hold data pointers and are linked together for range queries.

B+ Tree (Order 3):
          [30 | 60]
         /    |    \
    [10|20] [40|50] [70|80]
       ↓       ↓       ↓
    Data     Data    Data

Search for 40:
  Root: 40 > 30, 40 < 60 → go middle child
  [40|50]: found! → access data block
  Total: 2 disk I/Os

Performance Comparison (1 Million Records)

MetricSingle-LevelMulti-Level (B+ Tree)
Disk I/Os (1M records)~15~4
Search timeO(log n) on indexO(logp n) — much faster
Index updateCostly reorganizationSelf-balancing
Range queriesSlowVery fast (linked leaves)
Storage overheadLow-mediumMedium
Practical useSmall-medium DBsAll large production DBs
Conclusion: Single-level indexing is simple and suitable for small databases. For large-scale systems with millions of records, multi-level indexing (especially B+ Trees) is far more efficient, offering O(logp n) search with minimal disk I/Os. All modern RDBMS (MySQL, PostgreSQL, Oracle) use B+ Trees as their default indexing mechanism.
Q10 SQL Queries Using IN, LIKE, BETWEEN, and EXISTS Operators 16 Marks

1. IN Operator

Checks if a value matches any value in a list or subquery.

-- Basic IN
SELECT * FROM Employee WHERE Dept IN ('IT', 'Finance');

-- NOT IN
SELECT Name, Dept FROM Employee
WHERE City NOT IN ('Chennai', 'Mumbai');

-- IN with Subquery
SELECT Name, Salary FROM Employee
WHERE Dept IN (
    SELECT DeptName FROM Department WHERE Budget > 250000

2. LIKE Operator

Pattern matching: % = zero or more characters, _ = exactly one character.

SELECT * FROM Employee WHERE Name LIKE 'A%';       -- starts with A
SELECT * FROM Employee WHERE Name LIKE '%l';       -- ends with l
SELECT * FROM Employee WHERE Name LIKE '%ra%';     -- contains 'ra'
SELECT * FROM Employee WHERE Name LIKE '___';      -- exactly 3 characters

Selects values within a given range — inclusive of both endpoints.

-- Salary range
SELECT Name, Salary FROM Employee
WHERE Salary BETWEEN 60000 AND 80000;

-- Date range

WHERE JoinDate BETWEEN '2019-01-01' AND '2021-12-31';

-- NOT BETWEEN
SELECT Name, Salary FROM Employee
WHERE Salary NOT BETWEEN 60000 AND 80000;

Returns TRUE if the subquery returns at least one row.

-- Employees in dept with budget > 400,000
SELECT E.Name, E.Dept
FROM Employee E
WHERE EXISTS (
    SELECT 1 FROM Department D
    WHERE D.DeptName = E.Dept AND D.Budget > 400000
);

-- NOT EXISTS: Depts with no high earners
SELECT DeptName FROM Department D
WHERE NOT EXISTS (
    SELECT 1 FROM Employee E
    WHERE E.Dept = D.DeptName AND E.Salary > 80000
);

Q11 Functional Dependency and Minimal Cover – E = {B→A, D→A, AB→D} 16 Marks

Introduction

A Functional Dependency (FD) X → Y means that for every value of X, there is exactly one value of Y. The Minimal Cover (Canonical Cover) is the smallest equivalent set of FDs with no redundancies.

Given: E = {B → A, D → A, AB → D}

Algorithm to Find Minimal Cover (3 Steps)

Step 1: Simplify Right-Hand Sides

Ensure each FD has a single attribute on the RHS.

B  → A  ✓  (already single)
D  → A  ✓  (already single)
AB → D  ✓  (already single)

Current set: {B → A, D → A, AB → D}  ← No change needed

Step 2: Remove Extraneous Attributes from LHS

Check each FD with multiple LHS attributes. Only AB → D qualifies.

Check if A is extraneous in AB → D:
  Remove A, check if B⁺ contains D using remaining FDs.

  Compute B⁺ using {B → A, D → A, AB → D}:
    Start: B⁺ = {B}
    B → A:  add A  →  B⁺ = {B, A}
    AB → D: B⁺ has both A and B → add D  →  B⁺ = {B, A, D}
    D → A:  A already in B⁺

  Final B⁺ = {A, B, D}
  Since D ∈ B⁺, A is EXTRANEOUS in AB → D
  Replace AB → D with B → D

Updated set: {B → A, D → A, B → D}

Step 3: Remove Redundant FDs

Check B → A:
  Remove B → A. Remaining: {D → A, B → D}
  Compute B⁺:
    B → D: add D  →  B⁺ = {B, D}
    D → A: add A  →  B⁺ = {B, D, A}
  A ∈ B⁺ → B → A is derivable → REDUNDANT → Remove it

Updated set: {D → A, B → D}

Check D → A: Remove it. Remaining: {B → D}
  Compute D⁺ = {D} — cannot reach A.
  A ∉ D⁺ → D → A is NOT redundant → Keep it.

Check B → D: Remove it. Remaining: {D → A}
  Compute B⁺ = {B} — cannot reach D.
  D ∉ B⁺ → B → D is NOT redundant → Keep it.

Final Minimal Cover

Minimal Cover of E = { B → D,  D → A }

Verification

  • B → D (direct)
  • B → A (via B → D → A) ✓ covers original B → A
  • D → A (direct) ✓ covers original D → A
  • AB → D: Since B → D, this is covered ✓

Step Summary

StepActionResult
InitialGiven FDs{B→A, D→A, AB→D}
Step 1Simplify RHSNo change needed
Step 2Remove extraneous A from AB→DReplace with B→D → {B→A, D→A, B→D}
Step 3Remove redundant B→A{D→A, B→D}
FinalMinimal Cover{B→D, D→A}
Conclusion: The minimal cover of E = {B→A, D→A, AB→D} is {B→D, D→A}. This is the most compact, non-redundant set of FDs that preserves all original dependencies. Minimal covers are essential in 3NF synthesis to minimize schema complexity.
Q12 NoSQL Models for E-Commerce – Key-Value, Document, Column-Family, Graph 16 Marks

Introduction

NoSQL (Not Only SQL) databases are non-relational databases designed for large-scale, flexible, high-performance storage. Unlike RDBMS, they do not require a fixed schema, making them ideal for unstructured or semi-structured data.

1. Key-Value Store

Structure: Data stored as simple key-value pairs. Examples: Redis, DynamoDB

Key: "cart:user123"
Value: {"items": [{"productID": "P001", "qty": 2}]}

Key: "session:abc456"
Value: {"userID": "U123", "loginTime": "2024-01-10 10:30"}

Best for: Session management, shopping carts, caching. O(1) lookups.

2. Document Store (Best for E-Commerce)

Structure: JSON/BSON documents. Examples: MongoDB, CouchDB

{
  "_id": "P001",
  "name": "Samsung Galaxy S24",
  "category": "Electronics",
  "price": 79999,
  "specifications": {"RAM": "8GB", "Camera": "50MP"},
  "tags": ["smartphone", "android", "5G"],
  "stock": 150
}
// MongoDB: Find electronics under Rs. 50,000
db.products.find({ category: "Electronics", price: { $lt: 50000 } })

// Average rating for a product
db.reviews.aggregate([
  { $match: { productID: "P001" } },
  { $group: { _id: "$productID", avgRating: { $avg: "$rating" } } }

3. Column-Family Store

Structure: Rows with column families. Examples: Apache Cassandra, HBase

Row Key: "P001"

Q13 Examine the Techniques Used for Deadlock Resolution in Databases 16 Marks

Introduction

A deadlock occurs when two or more transactions wait indefinitely for each other to release locks, creating a cycle of dependencies where no transaction can proceed.

T1: Lock(A) → Request Lock(B) → WAITING for T2
T2: Lock(B) → Request Lock(A) → WAITING for T1
→ Neither can proceed. Deadlock!

Deadlock Prevention Techniques

Ensure that the conditions for deadlock can never arise.

1. Wait-Die Scheme (Non-Preemptive)

  • If Ti is older than Tj: Ti waits
  • If Ti is younger than Tj: Ti dies (rolls back and restarts)
T1 (older, ts=5) wants lock held by T2 (newer, ts=10) → T1 waits
T2 (newer, ts=10) wants lock held by T1 (older, ts=5) → T2 dies

2. Wound-Wait Scheme (Preemptive)

  • If Ti is older than Tj: Ti wounds Tj (forces Tj to rollback)
  • If Ti is younger than Tj: Ti waits

3. Resource Ordering

Assign a global ordering to all resources. Transactions must request resources in that fixed order — eliminates circular wait.

Deadlock Detection Techniques

Allow deadlocks to occur, then detect and recover.

1. Wait-For Graph

  • Nodes = Transactions
  • Edge Ti → Tj means Ti is waiting for Tj to release a lock
  • A cycle in the graph = Deadlock detected
Wait-For Graph:
T1 → T2 → T3 → T1   ← Cycle! Deadlock detected.

Algorithm:
  1. Maintain wait-for graph
  2. Periodically run DFS cycle detection
  3. Cycle found → select victim → rollback

2. Timeout-Based Detection

If a transaction waits longer than a threshold → assume deadlock → rollback. Simple but may rollback non-deadlocked transactions.

Deadlock Recovery – Victim Selection

  • Minimum rollback cost (least work done)
  • Minimum number of locks held
  • Oldest or newest transaction (policy-based)
  • Lowest priority transaction

Starvation Prevention: Track rollback count — avoid repeatedly selecting the same victim.

Comparison Table

TechniqueTypeApproachOverheadUse Case
Wait-DiePreventionTimestampLowHigh contention
Wound-WaitPreventionTimestampMediumHigh contention
Resource OrderingPreventionFixed orderLowPredictable access
Wait-For GraphDetectionGraph cycleMediumGeneral purpose
TimeoutDetectionTimerVery LowSimple systems
Conclusion: Deadlock is an inherent challenge in concurrent database systems. Prevention (Wait-Die, Wound-Wait, Resource Ordering) avoids deadlocks before they occur. Detection (Wait-For Graph, Timeout) identifies and resolves them after. Modern RDBMS like MySQL and PostgreSQL use wait-for graph detection with victim rollback.
Q14 Log-Based Recovery Techniques in DBMS 16 Marks

Introduction

Recovery management restores the database to a consistent state after failures (crash, power failure, disk errors). Log-based recovery maintains a sequential record of all modifications, enabling redo and undo of changes.

Types of Failures

  • Transaction failure – Logic error or deadlock
  • System crash – OS failure, power outage
  • Disk failure – Physical damage to storage

The Log File

An append-only file recording every database operation before it is applied. Format: <Transaction, Item, Old Value, New Value>

<T1, Start>
<T1, A, 500, 300>    -- T1 changes A from 500 to 300
<T1, B, 200, 500>    -- T1 changes B from 200 to 500
<T1, Commit>         -- T1 committed successfully
<T2, Start>
<T2, C, 1000, 800>
<T2, Abort>          -- T2 aborted

Write-Ahead Logging (WAL) Protocol

Rule: Before any data block is written to disk, its log record must be written to stable storage first. This ensures enough information to redo or undo changes after a crash.

Recovery Techniques

1. Deferred Update (No-Undo / Redo)

Database not updated until transaction commits. On failure: ignore uncommitted transactions (no undo needed). On recovery: redo committed transactions from log.

<T1, Start>
<T1, A, -, 300>    -- deferred (old value not stored)
<T1, Commit>       -- NOW apply A=300 to database

Crash before commit → A not changed → No undo needed

2. Immediate Update (Undo / Redo)

Database updated as operations execute. Log includes old and new values. On failure: undo uncommitted, redo committed transactions.

Log at crash time:
  <T1, Start>
  <T1, A, 500, 300>  ← T1 changed A
  <T1, Commit>       ← T1 committed ✓
  <T2, Start>
  <T2, B, 200, 500>  ← T2 changed B
  *** CRASH ***       ← T2 not committed ✗

Recovery:
  REDO T1: Set A = 300
  UNDO T2: Set B = 200  (restore old value)

Checkpoint-Based Recovery

Periodically flush dirty pages to disk and write a checkpoint record — avoids scanning the entire log on recovery.

<T1, Commit>
<CHECKPOINT {T2}>   -- T1 already safe on disk
<T2, Start>
<T2, B, 200, 500>
*** CRASH ***

Recovery: Start from checkpoint
  T2 active at checkpoint → UNDO T2
  T1 already checkpointed → No redo needed

ARIES Recovery Algorithm

Used by DB2, SQL Server, PostgreSQL. Three phases:

  1. Analysis Phase: Scan from last checkpoint → find dirty pages and active transactions at crash time
  2. Redo Phase: Redo all logged operations to bring database to crash state
  3. Undo Phase: Undo operations of all transactions that were active (not committed) at crash

Summary

TechniqueUpdates AppliedUndo NeededRedo Needed
Deferred UpdateAfter commitNoYes
Immediate UpdateBefore commitYesYes
Shadow PagingAfter commitNoNo
ARIESBefore commitYesYes
Conclusion: Log-based recovery is the industry standard for database durability. WAL ensures logs are always ahead of actual writes. Checkpoints reduce recovery time. ARIES provides a robust three-phase recovery mechanism used in most modern production databases.
Q15 RAID Levels – Comparison Based on Redundancy, Performance, and Cost 16 Marks

Introduction

RAID (Redundant Array of Independent Disks) is a storage technology combining multiple physical drives into one logical unit to improve performance, reliability, or both. Different RAID levels offer different trade-offs.

RAID 0 – Striping (No Redundancy)

Data is split across multiple disks in blocks. Reads and writes happen in parallel.

Data: [A1][A2][A3][A4]
Disk 1: [A1][A3]
Disk 2: [A2][A4]
  • Redundancy: None — one disk failure loses all data
  • Performance: Read & Write: Excellent
  • Storage Efficiency: 100%
  • Min Disks: 2
  • Use Case: Video editing, gaming

RAID 1 – Mirroring

Data duplicated across two disks identically.

Disk 1: [A1][A2][A3]  (original)
Disk 2: [A1][A2][A3]  (mirror)
  • Redundancy: Excellent — survives 1 disk failure
  • Performance: Read: Good | Write: Slower
  • Storage Efficiency: 50%
  • Use Case: OS drives, critical databases

RAID 5 – Striping with Distributed Parity

Data striped; parity distributed across all disks. Parity enables recovery from one disk failure.

Disk 1: [A1][B1][C1][P_D]
Disk 2: [A2][B2][P_C][D2]
Disk 3: [A3][P_B][C3][D3]
Disk 4: [P_A][B4][C4][D4]
  • Redundancy: Good — survives 1 disk failure
  • Storage Efficiency: (n-1)/n × 100% — e.g., 75% for 4 disks
  • Min Disks: 3
  • Use Case: File servers, NAS systems

RAID 6 – Double Distributed Parity

Like RAID 5 but with two parity blocks per stripe — survives two simultaneous disk failures.

  • Redundancy: Excellent — survives 2 failures
  • Storage Efficiency: (n-2)/n × 100%
  • Min Disks: 4
  • Use Case: Archives, cloud storage, hospitals

RAID 10 (1+0) – Mirroring + Striping

Combines RAID 1 (mirroring) and RAID 0 (striping) for maximum performance and redundancy.

RAID 0 stripe →  [Mirror Pair 1]    [Mirror Pair 2]
                  Disk1 | Disk2      Disk3 | Disk4
                   A1   |  A1         A2   |  A2
  • Redundancy: Excellent — multiple failures (one per pair)
  • Performance: Read & Write: Excellent
  • Storage Efficiency: 50%
  • Min Disks: 4
  • Use Case: High-performance databases, OLTP
